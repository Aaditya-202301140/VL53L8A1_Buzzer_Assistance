# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Firmware for a **blind/obstacle assistance system**: a NUCLEO-F401RE reads a
VL53L8A1 multizone ToF sensor and drives a passive piezo buzzer (via a BD139
transistor) at a beep rate that increases as an obstacle gets closer. It also
speaks a line protocol over the ST-Link virtual COM port to a companion Python
GUI for live telemetry and configuration.

Hardware: NUCLEO-F401RE (STM32F401RETx) · VL53L8A1 ToF sensor · passive piezo
buzzer + BD139 · buzzer on **PB10 (TIM2_CH3, PWM)**.

## Build / flash

This is an STM32CubeIDE managed-build project. There is no top-level build
script; the makefiles under `STM32CubeIDE/Debug/` are generated by the IDE.

- Normal workflow: open `STM32CubeIDE/` in STM32CubeIDE and Build/Debug. The
  `.ioc` is `Blind_Assistance_System.ioc` (regenerating code from it overwrites
  the `Src/`/`Inc/` files below — see "Editing rules").
- Command-line build (needs "GNU Tools for STM32" + `make` on PATH):
  `make -C STM32CubeIDE/Debug all` → produces `53L8A1_SimpleRanging.elf`.
  `make -C STM32CubeIDE/Debug clean` to clean.
- Output artifact / project name is historically `53L8A1_SimpleRanging`
  (the example this was forked from), even though the `.ioc` is named
  `Blind_Assistance_System`. Don't be surprised by the mismatch.

There are no automated tests — this is bare-metal firmware; "testing" means
flashing the board and driving it with the sensor + serial console/GUI.

## Architecture

Execution is a classic CubeMX skeleton (`main.c`) delegating all app logic to
one file:

- `main.c` — clock/GPIO/TIM2 init, then `MX_TOF_Init()` once and
  `MX_TOF_Process()` forever. `htim2` (the buzzer PWM) is the one globally
  shared peripheral handle; `app_tof.c` reaches it via `extern`.
- **`Src/app_tof.c` — the entire application.** Everything meaningful lives
  here: sensor init/config, the main ranging loop, buzzer beep-rate logic, the
  UART line protocol, and the interrupt-driven RX path.
- `Src/stm32f4xx_it.c` — ISRs. Notably `USART2_IRQHandler` forwards to
  `HAL_UART_IRQHandler(&hcom_uart[COM1])`, which is what makes RX work at all.
- `Src/stm32f4xx_nucleo*.c`, `Src/app_tof_pin_conf.c` — BSP/pin glue (COM port,
  button, sensor power/LPn pins). The VL53L8CX driver stack is external, under
  the pack's `Drivers/BSP/`.

### Ranging → buzzer loop (`MX_53L8A1_SimpleRanging_Process`)

Configured for `RS_PROFILE_8x8_CONTINUOUS`. Each iteration scans **all zones**
and takes the single **minimum distance** across every zone with a target, then
maps that distance to a buzzer behavior using three runtime thresholds
(`dist_near` < `dist_far_1` < `dist_far_2`, defaults 500 / 1500 / 2500 mm):

- `< dist_near` → continuous tone (DANGER)
- `< dist_far_1` → fast beep, 150 ms toggle (WARNING)
- `< dist_far_2` → slow beep, 500 ms toggle (CAUTION)
- else → silent (SAFE)

Tone is produced by `__HAL_TIM_SET_COMPARE(&htim2, TIM_CHANNEL_3, 500)` (≈50%
duty on the 0–999 period → audible); `0` = silent. Never call
`HAL_TIM_PWM_Stop` for silence — just set compare to 0.

### UART line protocol (firmware ⇄ Python GUI)

Over USART2 / ST-Link VCP (`hcom_uart[COM1]`). This protocol is the contract
with the external GUI — **changing these strings breaks the GUI**:

- **TX (streamed every loop):** `DIST:<mm>,ZONE:<SAFE|CAUTION|WARNING|DANGER>,ALERT:<OFF|SLOW_BEEP|FAST_BEEP|CONTINUOUS>,STATUS:RUNNING\r\n`
- **RX commands (one per `\r`/`\n`-terminated line):**
  - `THR:<danger>,<warning>,<caution>` — live thresholds (mm). Rejected unless
    strictly increasing; on success updates the runtime vars and replies
    `ACK:THR,<d>,<w>,<c>\r\n`.
  - `SENSOR:<res>,<timing>,<freq>` — queues a reconfigure (`res` leading digit
    `4`/`8` → 4x4/8x8; timing clamped 5–100 ms; freq clamped 1–30 Hz), applied
    on the next loop via Stop→ConfigProfile→Start.
  - `CMD:START` / `CMD:STOP` / `CMD:BUZZ_TEST`.
  - Single chars `r` / `s` / `c` at line start — legacy terminal controls
    (toggle resolution / signal+ambient / clear screen), handled directly in
    the RX ISR.

### RX is interrupt-driven — do not regress it

The RX path is deliberately interrupt-based, and it is fragile in a specific
way that has been fixed several times (see the long comment blocks in
`app_tof.c` and `stm32f4xx_it.c`). If you touch UART, preserve **all** of:

1. `USART2_IRQHandler` in `stm32f4xx_it.c` must exist and call
   `HAL_UART_IRQHandler(&hcom_uart[COM1])`. Without it, bytes arrive on the
   wire but never reach software (the weak default handler eats them).
2. `UART_StartReceive_IT()` must `HAL_NVIC_EnableIRQ(USART2_IRQn)` — `BSP_COM_Init`
   configures the peripheral for polling and does **not** enable the NVIC line.
3. `HAL_UART_RxCpltCallback` must **re-arm** `HAL_UART_Receive_IT` for every
   byte, or RX stops after one byte.
4. `HAL_UART_ErrorCallback` must clear ORE/NE/FE/PE and re-arm, or a single
   overrun permanently wedges RX.

The ISR only assembles a line into `rx_line` and sets `rx_line_ready`; parsing
happens in the main loop (`Process_GUI_Line`) — never parse inside the ISR.
`get_key()` / `com_has_data()` are dead polling helpers kept only for link
compatibility; don't reintroduce them into the RX path.

## Editing rules

- App logic changes go in `Src/app_tof.c`. CubeMX-owned files (`main.c`,
  `stm32f4xx_it.c`, `*_hal_msp.c`) use `/* USER CODE BEGIN/END */` guards —
  keep custom code strictly inside those, since regenerating from the `.ioc`
  wipes everything outside them.
- `STM32CubeIDE/Debug/` contains generated build output (`.o`, `.elf`, `.map`,
  `.su`, `.cyclo`, makefiles). Don't hand-edit it; it's rebuilt by the IDE.
